# Brook for GPUs - 笔记总结

## 核心思想
Brook 是一个将 GPU 抽象为**流处理器**的编程系统，通过扩展 C 语言提供数据并行编程模型，让开发者无需深入图形 API 就能利用 GPU 进行通用计算。

---

## 主要贡献

### 1. 流编程模型
- **Stream（流）**：同类型数据的集合，支持多维（如 `float s<10,5>`）
- **Kernel（内核）**：应用于流每个元素的并行函数
- **Reduction（归约）**：将流元素合并为单个值或更小的流（需满足结合律）
- 支持**用户自定义结构体**类型（如 Ray 结构）
- 区分**输入流**（顺序访问）和 **Gather 流**（随机访问/纹理读取）

### 2. 硬件抽象与虚拟化
- **流 → 纹理**：将流映射为 GPU 纹理，自动处理数据上传/下载
- **多输出支持**：当硬件输出数量不足时，自动将内核拆分为多 Pass
- **超维度/超大流**：通过地址转换逻辑，支持超过硬件纹理尺寸限制的流（1D 流从 2048 扩展到 131072 元素）
- **跨平台**：支持 OpenGL、DirectX 和 CPU 参考实现

### 3. 性能分析模型
定义**计算强度**（Computational Intensity）：
```
γ = K_gpu / T_r  （内核执行时间 / 数据传输时间）
```
GPU 优于 CPU 的条件：
```
γ > 1 / (s - 1)   其中 s = K_cpu / K_gpu
```
**结论**：计算强度高、单次调用工作量大的算法更适合 GPU。

## 实现技术要点

| 组件     | 实现方式                                |
| -------- | --------------------------------------- |
| 编译器   | 基于 cTool，将 Brook 内核转为 Cg 着色器 |
| 运行时   | 管理纹理、着色器绑定、渲染调用          |
| 流存储   | 浮点纹理（结构体成员分存不同纹理）      |
| 内核执行 | 渲染与输出流大小相同的四边形            |
| 归约     | 多 Pass 方法，O(log n) 复杂度           |

## 性能测试结果

测试应用：SAXPY、SGEMV（BLAS）、图像分割、FFT、光线追踪

| 应用     | GPU vs CPU 加速比 | 备注                                         |
| -------- | ----------------- | -------------------------------------------- |
| SAXPY    | **7x**            | 即使单条 MAD 指令也胜出，因 GPU 内部带宽更高 |
| 分割     | **4.7x**          |                                              |
| FFT      | ~1x（持平）       | CPU 版本（FFTW）利用缓存阻塞优化极佳         |
| 光线追踪 | 数倍于 CPU        | 报告射线-三角形相交测试率                    |

**观察**：
- DirectX 后端性能接近手写 GPU 代码（80% 以内）
- OpenGL 后端因 Pbuffer 拷贝开销，性能明显偏低
- ATI 浮点纹理带宽优于 NVIDIA（4.5 vs 1.2 Gfloats/sec），但 NVIDIA 峰值计算性能略高

## 关键权衡

### 数据传输开销
- `streamRead`/`streamWrite` 成本显著影响整体性能
- 只有**计算时间足以隐藏传输时间**时，GPU 才有优势
- 小规模数据受**内核启动开销**（CPU 设置时间）限制

### 内核拆分的代价
- 多输出拆分为单输出时，若输出间有共享计算，总指令数会增加（如 Cloth 示例中几乎翻倍）
- 但若输出可独立计算（如矩阵乘法），总运算量不变，仅纹理加载增加

---

## 局限性
1. **纹理精度限制**：地址计算依赖浮点精度（ATI 24-bit 限制为 131072 元素）
2. **无法支持**：ScatterOp/GatherOp 等间接读写操作（仍在 CPU 执行）
3. **归约需要多 Pass**：缺乏累加寄存器，内存访问量大于 CPU 实现
4. **OpenGL 渲染到纹理**：Pbuffer 切换开销大

---

## 对未来硬件的建议
1. **异步数据传输**：让读写与计算重叠
2. **统一内存**：消除 CPU-GPU 数据拷贝
3. **可变输出**：支持有条件输出零或多个元素
4. **顶点纹理 + 浮点混合**：支持更灵活的间接操作
5. **谓词执行**：利用深度测试等机制跳过无用计算

---

> **整体评价**：Brook 证明了将 GPU 视为流处理器的可行性，为后续 CUDA/OpenCL 等技术奠定了思想和实践基础。其核心价值在于**抽象硬件复杂度**，让非图形专家也能利用 GPU 计算能力。

# Z-buffer

深度测试的核心数学其实并不复杂，它主要解决两个问题：**“如何存储深度”** 和 **“如何比较深度”**。

---

### 1. 深度的数学定义

在三维空间中，深度（Depth）指的是顶点到**视点（Camera）** 在 **Z 轴方向**上的距离。

-   在**观察空间（View Space）** 中，物体在相机前方，Z 值是**负的**（OpenGL 约定看向 -Z 方向）。
-   为了方便存储，我们通过投影矩阵把它映射到一个 **0 到 1**（或 -1 到 1）的范围内，这就是 **NDC 深度**。

---

### 2. 从 3D 坐标到 Z-buffer 值的数学变换（透视投影）

这是最核心的数学公式。假设你在观察空间中有一个顶点的 Z 值（记为 $ Z_{view} $），它是一个**负数**。

#### 第一步：透视投影矩阵映射到 Clip Space

透视投影矩阵会把 $ Z_{view} $ 映射到 Clip Space 的 $ Z_{clip} $：

$$
Z_{clip} = - \frac{f + n}{f - n} \cdot Z_{view} - \frac{2fn}{f - n}
$$

- $ n $：近裁剪面距离
- $ f $：远裁剪面距离

#### 第二步：透视除法（Perspective Division）

GPU 硬件会自动把 $ Z_{clip} $ 除以 $ W_{clip} $（在透视投影中，$ W_{clip} = -Z_{view} $），得到 **NDC 深度 $ Z_{ndc} $**：

$$
Z_{ndc} = \frac{Z_{clip}}{W_{clip}} = \frac{Z_{clip}}{-Z_{view}}
$$

代入第一步的 $ Z_{clip} $，化简得：

$$
Z_{ndc} = \frac{f + n}{f - n} + \frac{2fn}{f - n} \cdot \frac{1}{Z_{view}}
$$

这个公式揭示了一个**关键特性**：$ Z_{ndc} $ 与 $ 1/Z_{view} $ 成正比（在 Vulkan 中为线性关系，在 OpenGL 中也为反比关系）。

#### 第三步：映射到 Z-buffer 值（Vulkan 范围 $[0, 1]$）

Vulkan 使用 $[0, 1]$ 范围，OpenGL 使用 $[-1, 1]$。这里以 Vulkan 为例：

$$
Z_{buffer} = 0.5 \cdot Z_{ndc} + 0.5
$$

---

### 🧠 这个数学公式告诉你什么？

**Z-buffer 中存储的 0~1 数字，和物体到相机的实际距离（$ Z_{view} $）不是线性关系！**

-   绝大多数精度（Bits）都分配给了**靠近近裁剪面（Near Plane）** 的区域。
-   距离相机越远，$ Z_{buffer} $ 值的变化越“拥挤”。

这就是 **Z-fighting（深度冲突）** 的数学根源：当两个物体离远平面很近时，它们的 $ Z_{buffer} $ 值极度接近，浮点数误差导致前后关系错乱。

---

### 3. 深度比较的数学逻辑

Z-buffer 本质上是一个二维数组，每个像素存储一个浮点数（通常是 24 位或 32 位）。

当光栅化生成一个片段时，硬件执行以下比较运算：

```cpp
// 伪代码：深度测试的逻辑
bool DepthTest(float fragmentDepth, int pixelX, int pixelY) {
    // 从 Z-buffer 中读取该像素已存储的深度值
    float storedDepth = zBuffer[pixelY * width + pixelX];
    
    // 根据当前设置的比较函数（默认是 LESS）做判断
    // LESS 表示：只有新片段更近（深度值更小）才通过
    if (fragmentDepth < storedDepth) {
        // 通过测试，后续会执行像素着色器（或直接写入）
        return true;
    } else {
        // 被遮挡，丢弃片段
        return false;
    }
}
```

### 4. 透视投影中 Z 的非线性分布（数值示例）

假设你的 **Near = 0.1**，**Far = 100.0**。

| 观察空间 $ Z_{view} $ | $ Z_{ndc} $ | Vulkan $ Z_{buffer} $ | 物理含义                          |
| :-------------------- | :---------- | :-------------------- | :-------------------------------- |
| -0.1（近平面）        | -1.0        | **0.0**               | 最近                              |
| -0.5                  | -0.6        | **0.2**               | 距离 5 倍，但 buffer 已经跳到 0.2 |
| -1.0                  | -0.2        | **0.4**               | 距离 10 倍                        |
| -10.0                 | 0.6         | **0.8**               | 远距离                            |
| -100.0（远平面）      | 1.0         | **1.0**               | 最远                              |

**观察**：

- 从 0.1 到 1.0（距离增加 10 倍），Z-buffer 值从 0 快速涨到 0.4。
- 从 10.0 到 100.0（距离又增加 10 倍），Z-buffer 值只从 0.8 涨到 1.0。

这就解释了为什么远处的物体更容易出现 Z-fighting（精度不够分了）。

---

### 5. 逆变换：从 Z-buffer 反推真实距离

如果你需要从深度贴图（Shadow Map）中还原世界坐标，你需要**反向**计算：

$$
Z_{view} = \frac{2fn}{f + n - Z_{ndc} \cdot (f - n)}
$$

这个公式在**阴影贴图（Shadow Mapping）** 中非常重要，用于比较当前片段与光源视角下的深度。

---

### 6. 数学带来的优化和注意事项

| 现象                       | 数学原因                                                     | 工程对策                                                     |
| :------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **Z-fighting（远处闪烁）** | 远平面附近的 Z 值变化太“钝”                                  | 减小 Far / Near 比例（如 Far/Near < 1000:1）；使用 **Reverse Z** 反转深度范围 |
| **Early Z 优化失效**       | 着色器写入了 `gl_FragDepth`，破坏了线性比较顺序              | 尽量避免写 `gl_FragDepth`                                    |
| **Reverse Z 技术**         | 将映射改为 $ Z_{ndc} = \frac{f}{f-n} - \frac{fn}{f-n} \cdot \frac{1}{Z_{view}} $，把近处精度移到远处 | 配合 `VK_COMPARE_OP_GREATER` 或 `VK_COMPARE_OP_LESS` 使用，显著提升远距离精度 |

---

### 7. 关于 Reverse Z 的数学

反向 Z 是现代引擎（如 DOOM 2016）广泛使用的技术，它通过**重新排列映射**，把浮点数的高精度区域分配给**远处**（实际应用中远处更容易出现精度问题）。

**常规 Z**（Near 为 0，Far 为 1）：  
$$
Z_{buffer} = \frac{1/Z_{view} - 1/Near}{1/Far - 1/Near}
$$

**Reverse Z**：  
$$
Z_{buffer} = \frac{1/Z_{view} - 1/Far}{1/Near - 1/Far}
$$

-   近处映射到 `1.0`，远处映射到 `0.0`。
-   配合 **Greater** 深度比较（仅当新深度 > 旧深度才通过），使得远处的 Z 值分布更密集，有效消除远处的 Z-fighting。

---

### 💎 总结（一句话版）

**Z-buffer 数学 = 把一个非线性的 $ 1/Z $ 值，映射到浮点数（0~1）存储，然后用 `<` 或 `>` 比较大小，决定像素的遮挡关系。**
